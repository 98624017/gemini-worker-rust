# /admin 相册视角与图表折叠设计

日期: 2026-04-16
状态: 已确认，待实现

## 目标

在不新增服务端 API、不中断现有 `/admin` 诊断能力的前提下，为管理页补齐两个交互改动：

1. 图表区域默认折叠，避免首次进入时抢占主要浏览动线。
2. 增加一个独立的“相册视角”，用于以更精致、更适合回看结果的方式浏览历史请求。

最终形态不是替换现有列表，而是在同一页内提供“列表视图 / 相册视图”切换：

- 列表视图继续承担排障、细节查看、错误分析职责。
- 相册视图承担回看、审阅、快速浏览生成结果职责。

## 约束

- 保持现有单文件实现方式：页面仍内嵌在 `src/admin.rs` 的 HTML 常量中。
- 不新增外部依赖，不引入构建工具、CDN、外部字体。
- 不新增服务端 API，继续只使用现有 `/admin/api/logs` 与 `/admin/api/stats`。
- 相册视图完全基于现有 `AdminLogEntry` 字段在前端聚合渲染。
- 现有列表视图的功能和可诊断性不能退化。

## 已确认决策

### 1. 图表默认折叠

- 图表区域保留，但默认处于折叠状态。
- 页面首次进入时，用户先看到 header、stats、视图切换和主内容区。
- 图表通过单独的折叠开关展开，不移除原有四张图。
- 折叠状态需要持久化到 `localStorage`，避免用户每次刷新都重复操作。

### 2. 相册视角采用“方案 B：胶片画廊”

相册视角整体气质采用更偏展示感的深色精致卡片布局，而不是继续沿用列表信息密集风格。原因：

- 现有列表视图已经承担完整的明细和排障职责。
- 相册视角的目标不是替代 detail panel，而是提高回看历史生成任务时的浏览效率和审美质量。
- 结果图在相册视角中应该比请求体图片更突出，形成清晰主视觉。

### 3. 分组方式按单次请求

相册数据按单次请求分组，一条 `AdminLogEntry` 对应一个相册组，不按日期或 prompt 主题再做二次归档。

这意味着相册区的时间顺序与当前日志列表一致，默认按时间倒序显示。

### 4. prompt 位置固定在请求图上方

每个相册组内，提示词区域必须位于请求体图片区的上方，不放到结果图下面，也不折叠到详情里。

提示词是本次请求的语义入口，应该先于图片被阅读。

### 5. 相册跳回现有列表中的对应记录

相册视图中的每个组都提供“查看对应记录”操作。点击后：

1. 自动切回列表视图。
2. 定位到该条日志记录。
3. 自动展开该日志的 detail 区域。
4. 给该条记录一个短暂的高亮提示，帮助用户建立视觉关联。

这条链路是本设计的重要要求，用来保证“精致浏览”和“深度诊断”可以无缝切换。

## 页面结构

### 顶部区域

顶部区域继续保留当前 header、状态信息、刷新与筛选能力。在此基础上新增两个控制点：

- 图表折叠开关
- 视图切换器：`列表视图` / `相册视图`

视图切换器应位于主操作区，具有明确选中态，且移动端仍易于点击。

### 图表区域

图表区域变为可折叠 section：

- 默认折叠
- 展开后仍显示原来的四张图
- 折叠时保留一个简洁标题和摘要文案，例如“4 个趋势图，按需展开”
- 展开/收起使用轻量动画，但必须兼容 `prefers-reduced-motion`

### 主内容区域

主内容区根据当前视图模式渲染不同布局：

- `list` 模式：沿用现有日志列表
- `album` 模式：渲染相册卡片组

两种模式共用同一份 `allItems` 数据、筛选条件、搜索条件和刷新结果。

## 相册视角设计

### 整体风格

视觉方向采用深色、高对比、低噪音、偏作品审阅的“胶片画廊”风格：

- 深色背景与深浅卡片层级
- 结果图作为主画幅
- 请求图作为辅图
- prompt 使用独立文案卡片承载
- 标签、元信息克制但清晰

不采用瀑布流，不做随意拼贴。整体应保持有秩序的卡片网格，以确保后台工具属性不丢失。

### 单个相册组结构

每条请求渲染为一个独立卡片，内部结构固定为：

1. 顶部元信息行
2. prompt 卡片
3. 请求体图片区
4. 结果图区
5. 底部操作区

#### 顶部元信息行

展示以下信息：

- 请求编号 `#id`
- 时间
- 状态码
- 耗时
- 可选标签：`stream`、`finishReason`、`slow`

要求：

- 信息密度高，但不能压过图片主体。
- 状态码和异常态要有明显区分。

#### Prompt 卡片

prompt 卡片位于请求图上方，作为该组的文字锚点。

要求：

- prompt 文案从现有 `requestRaw` 中做前端最佳努力提取，优先读取常见请求结构中的文本字段，例如 `contents[].parts[].text`。
- 如果无法稳定提取出 prompt，则退化为展示精简后的请求体文本片段，而不是留空。
- 默认显示主要内容，限制最大可见高度，避免长 prompt 把卡片撑坏。
- 提供“展开 / 收起”能力查看全文。
- 长文仍需保留换行和基本可读性。

#### 请求体图片区

请求体图片使用较小缩略图网格展示：

- 如果有多张图，按统一比例和圆角排列。
- 保留 cache 命中标记。
- 无请求图时，此区整体隐藏，不留空壳。

#### 结果图区

结果图区是主视觉区域：

- 结果图比请求图更大。
- 优先显示第一张结果图。
- 如果结果图有多张，主图外可补充次级缩略图，避免卡片高度无限增长。
- 没有结果图时，显示稳妥的 empty / error 状态，而不是留白。

#### 底部操作区

至少包含：

- `查看对应记录`

可选补充：

- 在新标签页打开主结果图
- 打开请求图原图

但这些是次要操作，不能盖过“跳回对应记录”。

## 状态与数据规则

### 过滤与搜索

当前已有的：

- status filter
- finishReason filter
- search

这些条件必须同时作用于列表视图和相册视图，保证用户切换视图时看到的是同一批结果，而不是两套不同数据状态。

### 视图状态持久化

以下状态建议写入 `localStorage`：

- 当前视图模式：`list` / `album`
- 图表是否展开

这样刷新页面后，用户会回到上次使用的浏览方式。

### 跳回列表定位规则

为每条列表项和相册卡片都绑定稳定的请求 ID。

从相册点击“查看对应记录”时：

1. 设置视图模式为 `list`
2. 等待列表渲染完成
3. 找到对应 `item.id` 的 DOM 节点
4. 如未展开，则自动展开 detail
5. 调用平滑滚动
6. 添加短时高亮态，约 1.5s 后消失

若目标项因筛选条件不可见，则优先保留当前筛选，不自动改筛选条件。此时不做隐式数据篡改，只提示或静默失败即可。为减少复杂度，首版可采用静默失败加 console 记录。

## 交互与动效

### 图表折叠动画

- 使用 `max-height + opacity` 实现轻量展开/收起
- 动效时长控制在 `150-220ms`
- `prefers-reduced-motion` 下应降级为无动画或近乎无动画

### 相册卡片 hover

- 轻微上浮
- 边框和阴影略增强
- 不使用夸张缩放

### 视图切换

- 切换器使用清晰的 active 态
- 切换时主内容区允许轻量淡入，但避免引入复杂页面级转场

### Prompt 展开

- 使用局部展开，不触发整页跳动
- 展开后的文本仍应控制可读宽度和行高

## 响应式要求

### 桌面端

- 相册使用双列或三列卡片网格，取决于可用宽度
- 每张卡片内部保持“prompt 在上、输入图与结果图成组”的层次

### 平板与窄屏

- 自动退为双列或单列
- 请求图缩略图尺寸随容器缩小，但触控区域仍满足可点击要求

### 手机端

- 相册退为单列
- 顶部元信息允许换行
- prompt 卡片与操作区保持清晰可点

## 无障碍与可用性

- 图标按钮必须有 `aria-label`
- 切换器和折叠按钮必须支持键盘操作
- 相册卡片中的图片应有有意义的 `alt` 文案，至少区分“请求图片”和“结果图片”
- 选中态、焦点态、错误态都要可见
- 交互尺寸满足最小点击热区

## 实现拆分

实现阶段建议按以下结构组织前端代码：

1. 扩展页面状态
   - 增加 `viewMode`
   - 增加 `chartsCollapsed`
   - 增加 `pendingScrollTargetId`

2. 扩展 DOM 结构
   - 增加图表折叠容器
   - 增加视图切换器
   - 增加相册容器

3. 拆分渲染函数
   - `renderList(items)` 继续负责列表
   - `renderAlbum(items)` 负责相册卡片
   - `renderMainContent(items)` 根据视图模式选择渲染函数

4. 增加跳转逻辑
   - `jumpToListItem(id)`

5. 将现有 refresh / filter / search 流程改为统一驱动两个视图

## 验证标准

实现完成后，至少验证以下场景：

1. 图表首次进入默认折叠，展开后能正常渲染四张图。
2. 相册视图可切换，刷新后保留上次视图状态。
3. 有请求图、有结果图的普通记录，显示结构正确。
4. 有 prompt 但无请求图时，请求图片区正确隐藏。
5. 无结果图或错误记录时，相册卡片有合理退化表现。
6. 点击“查看对应记录”能切回列表并定位到对应日志。
7. 搜索、状态筛选、finishReason 筛选在列表和相册中表现一致。
8. 移动端宽度下，相册布局能退化为单列且无横向滚动。

## 非目标

本次不做以下扩展：

- 不新增独立 `/admin/album` 路由
- 不引入服务端聚合接口
- 不做按日期分组或按 prompt 聚类
- 不做图片灯箱、全屏看图或批量导出
- 不重写现有日志 detail 结构

## 推荐落地顺序

1. 先完成图表折叠与视图切换状态管理
2. 再完成相册卡片渲染
3. 再接入“跳回对应记录”
4. 最后做样式打磨与响应式修正
