# Proxy Extra Overhead Design

**目标**

在不明显抬高峰值内存的前提下，尽量降低请求经过 `rust-sync-proxy` 后由本层额外增加的耗时，重点关注带图片处理的请求。

**问题定义**

这里优化的不是上游模型本身的耗时，也不是外部图床天然的网络时间，而是：

`经过本层的总耗时 - 直连上游的总耗时 = 本层额外增加的耗时`

目标是把这部分额外耗时尽量压低。

**硬约束**

1. 不允许明显反弹峰值内存
2. 不允许牺牲稳定性和正确性
3. 不为了“更快”而把请求侧专属逻辑污染到响应侧
4. 任何优化都必须同时报告耗时收益和内存代价

**优化范围**

本次设计覆盖本层增加耗时的 5 个阶段：

1. 请求接入与解析
2. 请求侧图片准备
3. 上游请求构造
4. 响应侧结果处理
5. `output=url` 上传阶段

**场景优先级**

优化重点放在带图片处理的请求，尤其是两条主路径：

- 图片重路径 `base64`
- 图片重路径 `output=url`

非图片请求仍纳入观测，但不是首轮优化主战场。

**指标设计**

必须同时建立两类指标：

1. **额外耗时指标**
   - 总额外耗时 `P50/P95`
   - 各阶段耗时 `P50/P95`
   - 缓存命中 / 未命中的分场景耗时
2. **内存代价指标**
   - 峰值 RSS
   - 热内存占用
   - spill 次数
   - spill 字节量

**分阶段观测点**

建议为每个请求记录以下时间戳或阶段耗时：

1. `request_parse_ms`
   从收到请求到 JSON / body 解析完成
2. `request_image_prepare_ms`
   请求侧 URL 拉图、缓存命中、必要格式转换
3. `upstream_build_ms`
   本层请求改写、编码、blob materialize、发起上游请求前准备
4. `upstream_wait_ms`
   等待上游响应返回
5. `response_process_ms`
   响应侧 JSON 归一化、挑图、base64 / URL 改写
6. `upload_ms`
   `output=url` 时的上传额外耗时
7. `proxy_overhead_ms`
   本层总额外耗时

**观测方式**

第一轮先选最小侵入方案：

- 在现有 `AdminLogEntry` / stats 链路上补充分阶段字段
- benchmark 脚本同时跑“直连上游”和“经过本层”两组样本
- 通过相同输入求差值得到本层额外耗时

这样可以复用现有 benchmark 基础，而不是另起一套工具。

**优化策略**

优先级按以下顺序执行：

1. 减少不必要的工作
   - 快路径绕开慢路径
   - 少做一次 JSON 扫描 / header 扫描 / 重复判断
2. 减少重复搬运
   - 少一次大 bytes 物化
   - 少一次 encode / decode
   - 少一次中间态构造
3. 减少同步等待
   - 缩短缓存命中路径
   - 明确哪些步骤必须同步，哪些不需要
4. 最后才考虑更复杂的流水线或并发化

**首轮优化候选**

首轮只做“内存近似不变但可能明显降低额外耗时”的改动：

1. 快路径裁剪
   - 非图片请求、轻图片请求尽量绕开重处理链
2. 请求侧缓存命中路径瘦身
   - 缩短命中缓存后的读出、封装、传递路径
3. 请求编码/改写路径减拷贝
   - 检查大对象在 `Value` / `Bytes` / `BlobHandle` 间的重复搬运
4. `output=url` 上传前准备减重
   - 优先减少上传前 materialize / 复制 / 再编码
5. 响应侧 JSON 处理单遍化
   - 合并重复扫描、避免多轮深拷贝

**第二轮优化候选**

只有在第一轮 benchmark 证明瓶颈仍明显存在时，才考虑：

- 更激进的流式化
- 阶段重叠
- 特殊上游专门快路径
- 更复杂的缓存键 / 对象复用策略

这些方案可能增加复杂度，甚至带来内存代价，不作为首轮默认项。

**收益判断规则**

所有优化方案统一按以下规则判断：

1. **直接接受**
   - `P95` 本层额外耗时明显下降
   - 峰值内存和热内存几乎不变
2. **单独评估**
   - `P95` 额外耗时大幅下降
   - 但峰值内存有可见上涨
3. **直接拒绝**
   - 额外耗时只小幅下降
   - 却把峰值内存明显抬高

也就是说，默认不是“拿内存换速度”，而是先追求：

**低额外耗时 + 峰值内存基本不反弹**

**基线样本**

第一轮 benchmark 统一固定四组样本：

1. 图片重路径，缓存未命中，`base64`
2. 图片重路径，缓存命中，`base64`
3. 图片重路径，缓存未命中，`output=url`
4. 图片重路径，缓存命中，`output=url`

所有优化都必须在同一组样本上复测，避免“换了输入，结果看起来更好”。

**预期产出**

本设计最终应交付：

1. 可重复的额外耗时 benchmark 基线
2. 分阶段耗时观测
3. 第一轮低内存代价优化包
4. 优化前后对比报告

**不做的事**

本轮不以“吞吐量极限”作为主目标，也不优先处理上游模型本身的耗时问题。核心仍然是：

**压低本层自己制造出来的额外耗时。**
